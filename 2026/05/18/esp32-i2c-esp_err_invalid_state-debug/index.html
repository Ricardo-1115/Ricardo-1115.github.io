ESP32-S3 I2C 踩坑记：ESP_ERR_INVALID_STATE 的排查与根因

发表于 2026-05-18 分类于 学习

在一块新焊接的 ESP32-S3 主控板上，PCA9685 舵机驱动初始化失败，abort() 报错：

问题现象

关键信息：

  • 同一套代码，在旧板上正常运行
  • 新板是自行焊接的 ESP32-S3 主控板，PCA9685 模块没换
  • 万用表测过 I2C 引脚（GPIO4/GPIO5）连通性：物理连接没问题
  • 外置 1kΩ 上拉电阻已焊接

排查过程

第一步：怀疑速度太快

既然是全新焊接的板子，首先怀疑 I2C 信号完整性。将 I2C 时钟从 400kHz 降到 100kHz：

结果：仍然报错。 不是速度问题。

第二步：I2C 地址扫描

既然设备存在且连线正确，上 I2C 扫描仪功能，看总线上到底有哪些设备：

输出：

意外发现一个 0x70 的神秘设备！查资料发现 0x70 是 PCA9685 的 ALLCALL 广播地址（MODE1 寄存器 Bit 5 默认开启），所以不是额外设备，虚惊一场。

但更重要的是：扫描之后 PCA9685 初始化居然成功了！

第三步：定位——是扫描本身起了作用

去掉扫描代码 → 回到原始代码 → 重新报错。反复验证确认：是扫描中的 i2c_master_probe 调用让 PCA9685 初始化成功，不是扫描的副产物。

分步定位：

测试 操作 结果
1 纯延迟 50ms（代替 probe） ❌ 失败
2 纯延迟 500ms ❌ 失败
3 probe 在 add_device 之前 ✅ 成功
4 probe 在 add_device 之后、写寄存器之前 ✅ 成功

结论：不是上电时序问题（延迟无效），而是 I2C 控制器在添加设备后、第一次数据传输前需要一次”预热”事务。

第四步：根因——trans_queue_depth = 0

注意到 trans_queue_depth = 0，这在 ESP-IDF v5.x 中意味着不为 I2C 驱动分配内部传输队列。查阅 ESP-IDF 文档和源码发现：

  • trans_queue_depth = 0 时，驱动使用”直接模式”处理事务
  • 在某些芯片上（芯片个体差异），第一次 i2c_master_transmit 时驱动内部状态机尚未完全就绪，返回 ESP_ERR_INVALID_STATE
  • i2c_master_probe 发送了一个空事务（START + Address + STOP），无意中”激活”了驱动内部状态
  • 设为 trans_queue_depth = 4 后，驱动在总线创建时就分配好队列资源，第一次传输正常

根因总结

项目 说明
直接原因 trans_queue_depth = 0 导致 I2C 驱动无内部队列，首次传输在某些芯片上状态异常
芯片差异 旧板 ESP32-S3 无此问题，新板有此问题，属芯片个体差异
为什么 scan 能 work i2c_master_probe 的空事务无意中”预热”了驱动状态机
为什么延迟没用 不是电源时序问题，是驱动层状态机的问题

教训

  1. trans_queue_depth = 0 是一个危险值。 即使它在大部分芯片上工作，也无法保证在所有芯片上可靠。设一个合理的队列深度（如 4）是更稳健的做法。
  2. I2C 扫描不仅是调试工具。 在这次排查中，扫描本身无意间成了”修复”手段，反过来帮助定位了根因。
  3. 万用表测连通性 ≠ 硬件没问题。 很多嵌入式问题出在芯片内部状态、驱动配置等软件层面，不要过早排除软件原因。
  4. 替换法（新旧板对比）是高效的手段。 能快速将问题范围缩小到”相同代码、不同硬件”的差异点上。
